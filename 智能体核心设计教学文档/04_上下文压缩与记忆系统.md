# 上下文压缩与记忆系统

记忆系统是这个项目最值得学习的部分之一。

大模型的上下文不是无限的。任务一长，消息、工具结果、文件内容、命令日志都会不断累积。如果不处理，模型很快就会“装不下”之前的信息。

本项目用了多层策略来解决这个问题。

## 为什么需要记忆

可以把模型上下文理解成一张临时办公桌。

桌子上可以放当前任务资料，但空间有限。旧资料如果全部堆在桌上，会影响当前工作。合理做法是：

- 当前正在用的资料放桌上。
- 之前的重要结论整理进笔记。
- 旧的详细过程归档。
- 不重要的长日志只保留摘要。

这个项目就是这么做的。

## 第一层：工具结果裁剪

在 `compression.py` 中，`apply_tool_result_budget` 会处理过长的工具结果。

如果某个工具输出超过设定长度，它只保留：

- 开头一段。
- 结尾一段。
- 中间用 `clipped` 标记省略。

为什么保留头尾？

因为命令输出和日志里，开头通常有命令信息，结尾通常有错误或最终结果。这比随机截断更有用。

## 第二层：旧工具结果摘要

`microcompact` 会把较早的工具结果改成简短摘要，但保留最近几条工具结果原文。

设计逻辑是：

- 最近工具结果通常还在当前推理链路里，要保真。
- 很早的工具结果多数只是历史证据，可以摘要。

这是一种低成本、每轮都能做的压缩。

## 第三层：session memory 结构化记忆

`session_memory.py` 维护了 `.memory/session_memory.md`。

这个文件不是随便写的，它有固定 section：

- `Session Title`
- `Current State`
- `Task specification`
- `Files and Functions`
- `Workflow`
- `Errors & Corrections`
- `Codebase and System Documentation`
- `Learnings`
- `Key results`
- `Worklog`

这相当于给智能体准备了一份标准化工作笔记模板。

## 为什么记忆要结构化

如果只让模型“总结一下之前内容”，结果很可能忽略重要信息。

结构化记忆强制模型从几个角度整理：

- 当前做到哪了。
- 用户到底要什么。
- 哪些文件和函数重要。
- 走过什么流程。
- 失败过什么，怎么修正的。
- 学到了什么。
- 已经交付了什么。

这比自由摘要更稳定。

## 记忆不是每轮都更新

`maybe_schedule_extraction` 会判断什么时候值得更新记忆。

首次初始化需要上下文达到一定 token 数。初始化之后，需要新增内容达到一定量，并且满足：

- 新增工具调用足够多，或
- 到达自然停顿点。

这样做是为了避免每轮都调用模型总结，浪费成本，也避免在工具调用链还没闭合时切碎上下文。

## 什么是自然停顿点

自然停顿点指最后一条 assistant 消息没有工具调用。

如果最后一条 assistant 正在调用工具，还没收到工具结果，这时不能推进记忆边界。

原因是工具调用和工具结果是一对。如果只总结了前半段，后半段留在原文里，后续压缩时上下文链可能断掉。

## 后台提取记忆

记忆提取是在后台线程里做的。

主循环不会傻等模型总结完，而是继续推进任务。只有真正要压缩时，才会短暂等待后台提取完成。

这是一个重要设计：

- 平时异步维护记忆，不阻塞主任务。
- 压缩前尽量等一下，使用最新记忆。
- 如果后台卡太久，就放弃等待，避免主流程被拖死。

## 压缩时发生什么

当上下文接近阈值时，主循环会调用 `auto_compact`。

优先策略是 session memory 压缩：

1. 确保消息都有 UUID。
2. 等后台记忆提取完成。
3. 检查 memory 文件是否有真实内容。
4. 找到已经被记忆覆盖的消息边界。
5. 保留最近一段原始消息。
6. 确保 tool call 和 tool result 不被拆散。
7. 把旧历史替换成一条 session memory 消息。
8. 加一个 compact boundary 标记。

压缩后的上下文大致变成：

```text
compact boundary
session memory summary
recent raw messages
```

这就像把桌上旧资料整理进笔记，只留下当前还要用的几页原文。

## 为什么要保留 recent raw messages

记忆文件是总结，可能丢细节。

最近的原始消息通常包含：

- 当前正在编辑的精确代码。
- 最新工具结果。
- 刚刚发生的错误。
- 用户最近的具体要求。

这些信息不能只靠总结，所以压缩后仍然保留最近原文。

## compact boundary 的作用

压缩后会插入一条 `compact-boundary` 系统消息。

它的作用是告诉后续压缩逻辑：

- 这里之前已经压缩过。
- 不要无限向前回扩。
- 不要把压缩过的内容又展开成庞大上下文。

这相当于在历史中打一个书签。

## fallback：legacy compact

如果 session memory 不可用，系统会退回旧版压缩：

1. 把完整历史写到 `.transcripts/`。
2. 调用模型临时总结整段历史。
3. 用这个 summary 替换旧上下文。

这不如结构化记忆稳定，但能保证系统不会因为 memory 失败而完全卡住。

## 记忆系统的核心设计思想

这个项目的记忆系统体现了几个成熟思想：

1. 不要等上下文爆了才处理，要提前压缩。
2. 旧工具结果可以摘要，近期结果要保真。
3. 长期记忆要结构化，而不是自由总结。
4. 记忆提取要异步，避免阻塞主任务。
5. 压缩边界要稳定，不能切断工具调用链。
6. 所有压缩策略都要有 fallback。
