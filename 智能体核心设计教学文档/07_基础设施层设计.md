# 基础设施层设计

`infra` 层负责底层技术能力。它不决定智能体要做什么，只提供可靠的执行接口。

## LLM 网关

`infra/llm.py` 是统一的大模型调用入口。

核心函数：

- `get_llm(max_tokens)`
- `invoke_langchain(messages, tools, system, max_tokens)`
- `render_ai_text(message)`
- `format_tool_guide(tools)`
- `estimate_tokens(messages)`

## 为什么要统一模型调用

项目支持多种模型：

- Qwen
- DeepSeek
- OpenAI 兼容接口

如果每个地方都自己写模型调用逻辑，后期很难维护。

统一放到 `llm.py` 后：

- 换模型只改配置或这一层。
- 工具绑定逻辑统一。
- system prompt 注入统一。
- AIMessage 文本抽取统一。

这就是适配器思想：把不同供应商差异包在一层里，上层只面对统一接口。

## 工具说明格式化

`format_tool_guide` 会把工具列表整理成模型可读文本。

它会提取：

- 工具名。
- 工具描述。
- 参数名。
- 参数类型。
- 参数是否必填。
- 参数说明。

这份工具说明会拼进 system prompt，让模型更清楚每个工具怎么用。

## token 估算

`estimate_tokens` 使用一个粗略方法估算上下文大小：

```text
把消息转成 JSON 字符串长度 / 4
```

这不是精确 tokenizer，但足够用于触发压缩策略。

这里体现了一个务实选择：触发阈值不需要绝对精确，只要能提前发现上下文膨胀趋势。

## Shell 执行层

`infra/shell.py` 负责运行命令。

它处理了两个问题：

1. Windows 下使用 PowerShell，并设置 UTF-8 输入输出编码。
2. 对明显危险命令做基础拦截。

危险命令包括：

- `rm -rf /`
- `sudo`
- `shutdown`
- `reboot`
- `> /dev/`

这个拦截不是完整沙箱，但能挡住一些明显错误操作。

## 为什么 shell 要封装

如果工具层直接调用 `subprocess.run`，会出现很多重复问题：

- 当前工作目录不统一。
- Windows 编码乱码。
- 超时设置不一致。
- 错误处理分散。
- 危险命令拦截分散。

统一封装后，上层只调用 `run_bash` 或 `run_shell`。

## 工作区访问层

`infra/workspace.py` 负责文件路径和编码。

核心函数：

- `safe_path(path_text)`：把路径限制在工作区内。
- `read_text_auto(fp)`：自动尝试 UTF-8、UTF-8 BOM、GB18030、GBK。
- `relative_display(path)`：把路径尽量展示为相对路径。

## safe_path 的重要性

智能体能读写文件是一把双刃剑。

必须明确限制：只能在当前工作区内操作。

`safe_path` 会把用户或模型传入的路径和 `WORKDIR` 拼接，然后 resolve 成绝对路径，再检查它是否仍位于工作区内。

这是智能体文件工具的基本安全设计。

## 编码兼容

项目路径和文件内容中有中文。Windows 环境下很多旧文件可能不是 UTF-8，而是 GBK 或 GB18030。

`read_text_auto` 会依次尝试多种编码，减少读取中文文件时的乱码问题。

这是非常贴近真实开发环境的设计。

## config.py 的作用

`config.py` 集中管理：

- 工作目录。
- 模型名称。
- API base URL。
- `.team/.tasks/.memory` 等目录。
- token 阈值。
- 压缩参数。
- session memory 更新参数。
- 后台轮询参数。

配置集中化的好处是：调参数时不用到处找，也减少魔法数字散落在代码里。

## 基础设施层的核心设计思想

`infra` 层体现了几个原则：

1. 上层不要直接碰供应商细节。
2. 上层不要直接碰 shell 细节。
3. 上层不要直接相信路径。
4. 编码、超时、安全边界要统一处理。
5. 参数集中配置，方便调优。
