# 核心设计思想总结

这一篇不再按文件解释，而是总结这个项目最值得学习的智能体设计思想。

## 设计思想一：模型负责决策，工具负责行动

大模型擅长理解目标、规划步骤、判断下一步。

但真正的读文件、搜代码、改文件、运行命令，必须交给工具执行。

这个项目通过 `tooling.py` 把工具暴露给模型，再通过 `handlers` 把工具调用落到真实函数上。

这就是智能体系统的基本形态：

```text
模型决定做什么
工具执行怎么做
工具结果反馈给模型
模型继续判断下一步
```

## 设计思想二：上下文是资源，不是垃圾桶

很多初学者会把所有历史都塞给模型，以为信息越多越好。

这个项目的做法更成熟：

- 单条工具结果过长，裁剪。
- 较早工具结果，摘要。
- 长历史，沉淀进结构化记忆。
- 最近原始上下文，保留。
- 记忆不可用时，走 fallback。

它背后的原则是：上下文要服务当前决策，而不是无差别保存全部过程。

## 设计思想三：记忆要结构化

自由摘要容易遗漏重点。

本项目使用固定 section 的 session memory：

- 当前状态
- 用户任务
- 文件和函数
- 工作流
- 错误和修正
- 学习结论
- 关键结果
- 工作日志

这让记忆更像工程笔记，而不是聊天摘要。

## 设计思想四：边界比能力更重要

智能体有工具后能力变强，但风险也变大。

项目里有很多边界设计：

- 路径限制在工作区。
- shell 拦截明显危险命令。
- 子代理有步数限制。
- Explore 子代理默认没有写权限。
- 写文件后清理读缓存。
- 压缩时不切断 tool call 和 tool result。

成熟的智能体系统不是“什么都让模型做”，而是“在明确边界内让模型做事”。

## 设计思想五：复杂任务需要显式状态

不能指望模型永远记得当前计划。

所以项目设计了：

- Todo：当前请求内的短期清单。
- TaskManager：持久任务板。
- MessageBus：协作消息。
- Team config：队友状态。
- SessionMemoryState：记忆边界状态。

这些状态文件让系统可以跨轮、跨线程、跨代理协作。

## 设计思想六：多智能体不是多开模型

真正的多智能体协作需要：

- 角色。
- 任务分配。
- 消息通道。
- 共享任务板。
- 生命周期管理。
- 权限控制。

本项目里的 `subagent` 和 `teammate` 展示了两种不同协作模型：

- 临时子代理：短任务、轻状态、返回摘要。
- 持久队友：有身份、有 inbox、可 idle、可认领任务。

这比简单并发调用模型更接近真实协作系统。

## 设计思想七：系统要能降级

智能体系统里，任何环节都可能失败：

- 模型调用失败。
- 记忆提取失败。
- 工具执行失败。
- 文件编码异常。
- 搜索结果过大。
- 后台线程卡住。

项目里多处做了降级：

- session memory 不可用时走 legacy compact。
- 后台提取卡太久就不等。
- 文件读取尝试多种编码。
- 编辑失败时提示重新读取更小锚点。
- 工具异常转成错误文本反馈给模型。

降级能力决定了智能体能不能在真实环境中继续工作。

## 设计思想八：提示词也是工程接口

`prompting.py` 不是随便写一段 system prompt。

它把运行环境、技能、工具说明和操作规则都组织起来：

- 当前工作目录。
- 操作系统和 shell。
- 工具参数说明。
- 搜索优先于读取。
- 读取优先于编辑。
- 长任务使用任务系统。
- 后台任务如何处理。
- 团队协作如何发消息。

提示词在这里相当于“给模型看的 API 文档和操作规程”。

## 最终可以学到什么

如果你要从这个项目迁移设计经验，最重要的是这些：

1. 先搭一个清晰主循环。
2. 工具注册和工具执行要分离。
3. 文件、shell、模型调用都要封装。
4. 上下文从一开始就要设计压缩策略。
5. 长期记忆要结构化并持久化。
6. 子代理要限制权限和步数。
7. 多代理协作必须有任务、消息和状态。
8. 每个可能失败的环节都要能降级。

学会这些，比记住某个函数的语法更重要。
