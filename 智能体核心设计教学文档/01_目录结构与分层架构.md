# 目录结构与分层架构

本项目的目录结构体现了一个核心思想：智能体系统不要写成一团，而要把“流程编排、业务对象、底层能力、运行状态”分开。

## 顶层结构

`final_version_app` 下面主要有这些内容：

```text
final_version_app/
  application/
  domain/
  infra/
  tests/
  .memory/
  .tasks/
  .team/
  config.py
  runtime.py
  coordination.py
  shared.py
```

## application：应用编排层

`application` 是最核心的一层。它负责“智能体怎么工作”。

主要文件：

- `agent_loop.py`：主智能体循环，决定每一轮怎么调用模型、工具、记忆压缩。
- `tooling.py`：把各种工具注册给模型，并把工具名映射到真正执行函数。
- `workspace_ops.py`：实现读文件、搜索、写文件、编辑文件这些工作区操作。
- `compression.py`：处理工具结果裁剪、上下文折叠、旧版总结压缩。
- `session_memory.py`：结构化长期记忆系统，是本项目很重要的设计点。
- `subagent.py`：短生命周期子代理，适合一次性探索或局部工作。
- `team_runtime.py`：长期队友代理，支持收件箱、任务认领、空闲等待。
- `repl.py`：命令行交互入口。
- `prompting.py`：组装系统提示词。
- `container.py`：集中创建各类服务对象。

可以把 `application` 理解成公司的“运营流程”。

## domain：领域对象层

`domain` 放的是项目自己的业务概念。

主要文件：

- `todos.py`：短期 Todo 清单。
- `tasks.py`：持久任务板，任务会保存到 `.tasks/`。
- `skills.py`：技能加载器，读取 `skills/SKILL.md`。
- `messaging.py`：文件型消息总线，队友之间靠 inbox 通信。
- `background.py`：后台命令任务管理器。

可以把 `domain` 理解成公司内部的“工作制度”：任务怎么登记、消息怎么传、技能怎么加载。

## infra：基础设施层

`infra` 放底层技术能力。

主要文件：

- `llm.py`：统一调用大模型，支持 Qwen、DeepSeek、OpenAI 兼容接口。
- `shell.py`：统一执行 shell 命令，并做基础危险命令拦截。
- `workspace.py`：安全处理路径、自动识别文件编码、展示相对路径。

可以把 `infra` 理解成公司使用的“设备和底层接口”。

## 状态目录

项目运行时会写入一些状态文件：

- `.memory/`：长期记忆文件和记忆状态。
- `.tasks/`：持久任务板，每个任务一个 JSON 文件。
- `.team/`：团队配置和成员 inbox。
- `.team/inbox/`：每个智能体一个收件箱文件。

这些目录说明了一个重要思想：智能体不应该只依赖当前内存。长任务、并行任务、跨轮任务都需要持久化状态。

## 兼容层

`coordination.py` 和 `shared.py` 主要是兼容旧导入路径。

它们不承担新的核心逻辑，只是把重构后的新模块重新导出。这样旧代码即使还在从旧路径导入，也不至于立刻坏掉。

这是项目演进中常见的工程手法：重构内部结构时，尽量保持外部接口稳定。

## 分层的核心价值

这个目录结构背后的设计思想是：

- 主流程只关心“什么时候该做什么”。
- 工具层只关心“某个动作怎么执行”。
- 领域层只关心“任务、消息、技能这些业务对象怎么管理”。
- 基础设施层只关心“模型、shell、文件系统这些底层能力怎么封装”。

这样的系统更容易扩展。例如以后要换模型，只改 `infra/llm.py`；要新增工具，主要改 `application/tooling.py`；要改记忆策略，主要改 `application/session_memory.py` 和 `application/compression.py`。
