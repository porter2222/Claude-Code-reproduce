# 学习路线与项目总览

这份文档面向刚接触智能体项目的学习者。阅读时不用先理解 Python 语法，先把它当成一个“会使用电脑完成任务的虚拟员工系统”来理解。

## 一句话理解这个项目

`final_version_app` 是一个本地命令行智能体应用。

用户在终端里提出任务后，系统会让大模型进行思考。大模型不能直接改文件、读代码或运行命令，它必须通过项目提供的工具来行动。主循环负责反复执行：

1. 接收用户问题。
2. 把当前上下文、系统提示词、工具列表交给大模型。
3. 如果大模型要调用工具，就执行工具。
4. 把工具结果再放回上下文。
5. 继续让大模型判断下一步。
6. 如果大模型不再调用工具，就输出最终回答。

所以它不是一个普通聊天机器人，而是一个“模型 + 工具 + 记忆 + 协作”的智能体运行时。

## 项目里的四类核心对象

可以用公司来类比：

- 主智能体：项目经理，负责判断下一步该做什么。
- 工具：电脑里的具体操作能力，比如读文件、搜索、写文件、运行命令。
- 记忆系统：工作笔记，防止任务太长后忘记之前做过什么。
- 子代理和队友：临时外包人员或长期协作同事，可以分担搜索、实现、沟通等任务。

## 为什么要拆成这么多层

如果把所有逻辑都写在一个文件里，短期能跑，但很难维护。这个项目采用了比较清晰的分层：

- `application`：负责智能体工作流程，例如主循环、工具装配、记忆压缩、子代理。
- `domain`：负责业务概念，例如任务板、Todo、消息、技能、后台任务。
- `infra`：负责底层技术能力，例如调用模型、执行 shell、访问文件系统。
- `config.py`：集中管理运行参数和目录位置。
- `.memory/.tasks/.team`：保存运行过程中的状态。

这个设计的好处是：主循环不用关心文件怎么安全读取，工具层不用关心模型怎么调用，记忆系统不用关心用户输入来自哪里。

## 学习这个项目最重要的不是语法

这个项目最值得学习的是五个设计思想：

1. 智能体要通过工具行动，不能只靠模型“想象”。
2. 上下文是有限资源，长任务必须主动压缩和记忆。
3. 工具要有边界，文件读写和命令执行都要被封装。
4. 长任务要显式管理进度，不能只靠模型脑内记忆。
5. 协作能力不是魔法，本质是消息、任务、状态和多个运行循环的组合。
