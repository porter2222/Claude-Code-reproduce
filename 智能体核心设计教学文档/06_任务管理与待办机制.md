# 任务管理与待办机制

这个项目里有两种任务管理：

- `TodoManager`：短期待办。
- `TaskManager`：持久任务板。

它们解决的问题不同。

## TodoManager：当前任务内的清单

`domain/todos.py` 维护一个内存里的 Todo 列表。

每个 Todo 有：

- `content`：要做什么。
- `status`：pending、in_progress、completed。
- `activeForm`：正在进行时的动作描述。

它主要用于当前请求内部的短链路规划。

## Todo 的约束

`TodoManager` 有几个约束：

- 最多 20 条。
- 只能有一个 `in_progress`。
- 每条必须有内容。
- 每条必须有 `activeForm`。
- 支持中文状态别名，例如“待办”“进行中”“完成”。

这些约束很重要。

如果允许多个 `in_progress`，模型可能同时声称在做很多事，但实际上无法聚焦。只允许一个进行中事项，可以强迫智能体明确当前焦点。

## 主循环如何提醒 Todo

`agent_loop.py` 会记录模型连续多少轮没有调用 `TodoWrite`。

如果 Todo 中还有未完成事项，并且模型多轮没有更新 Todo，系统会插入提醒消息。

这是一种轻量的自我约束机制，避免模型列了计划后就忘。

## TaskManager：持久任务板

`domain/tasks.py` 维护 `.tasks/` 目录下的任务文件。

每个任务保存为：

```text
.tasks/task_{id}.json
```

任务包含：

- `id`
- `subject`
- `description`
- `status`
- `owner`
- `blockedBy`
- `blocks`

这比 Todo 更适合复杂、多步骤、跨代理协作的工作。

## 任务依赖设计

任务可以有：

- `blockedBy`：当前任务被哪些任务阻塞。
- `blocks`：当前任务完成后会解除哪些任务。

当某个任务被标记为 `completed` 时，系统会扫描其他任务，把它们的 `blockedBy` 中对应任务 ID 移除。

这就是一个很轻量的 DAG 任务图。

## claim_task 的作用

`claim_task` 会把任务分配给某个执行者，并把状态改成 `in_progress`。

主智能体会以 `lead` 身份认领任务；队友代理会以自己的名字认领任务。

这让任务不只是列表，而是可以被不同执行者分工。

## Todo 和 Task 的区别

| 维度 | Todo | Task |
|---|---|---|
| 保存位置 | 内存 | `.tasks/` 文件 |
| 生命周期 | 当前请求内 | 可跨轮存在 |
| 适合场景 | 简短计划 | 多步骤项目 |
| 协作 | 不适合多人分配 | 支持 owner 和依赖 |
| 复杂度 | 简单 | 更结构化 |

## 为什么两个都需要

如果只有 Todo，复杂任务缺少持久状态和依赖关系。

如果只有 Task，每个小步骤都落盘会显得笨重。

所以本项目采用双层任务管理：

- 小任务用 Todo 约束当前行动。
- 大任务用 TaskManager 组织协作和依赖。

这也是很多工程系统的通用模式：短期工作记在 checklist，长期项目放在任务系统。
