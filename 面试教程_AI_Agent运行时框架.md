### 面试话术

> "这个项目的架构是经典的三层分层设计：

> **第一层——基础设施层**：封装了与外部系统的交互，包括 LLM API 调用网关 (`infra/llm.py`)、Shell 命令执行引擎 (`infra/shell.py`)、以及工作区文件安全访问层 (`infra/workspace.py`)。这一层的核心设计原则是 **安全与兼容**——比如 Shell 执行会拦截危险命令，路径访问会防止越界，文件读取会兼容多编码格式。

## 第三步：核心流程——主循环讲透

### 🔄 这是面试中**最核心**的部分，一定要背熟！

```
                   ┌─────────────┐
                   │   用户输入    │
                   └──────┬──────┘

             ┌─────────────────────┐

             │  ├─轻量: 裁剪工具输出  │
             │  ├─中等: 摘要重写     │
             │  └─重度: 归档+重置    │
             └──────────┬──────────┘

             ┌─────────────────────┐

             └──────────┬──────────┘

             ┌─────────────────────┐
             │  4. 调用 LLM         │
             │  - 选模型 (deepseek) │
             │  - 发消息            │
             │  - 收响应            │
             └──────────┬──────────┘

             ┌─────────────────────┐
             │  5. 工具调用分支      │
             │  ┌─有tool_call?────┐ │
             │  │  Yes → 执行工具  │ │
             │  │  No  → 输出文本  │ │
             │  └────────┬───────┘ │
             │           ▼         │
             │    回到步骤2继续     │
             └─────────────────────┘
                        ▼
                   ┌─────────────┐
                   │  6. 结束输出  │
                   └─────────────┘
```

### 面试话术（核心必背）

> "整个系统的核心是 **`agent_loop.py` 里的主循环**，这是一个『推理-行动』循环 (ReAct Loop)，流程如下：

> **第一步，消息注入与标注**：把用户最新的输入追加到消息列表，给每条消息生成一个 UUID 用于后续追踪，同时把持久化的 Session Memory 注入到上下文里。

> **第二步，Token 预算检查**：这是保证长对话不炸内存的关键。我设计了**三级压缩策略**：
> - **轻量压缩**：裁剪单条超长的工具输出（比如文件内容超过 2400 字符就只保留头尾）
> - **中等压缩**：把较早的工具调用改写为摘要形式
> - **重度压缩**：当上下文超过 80% 阈值时，把历史归档到 `.transcripts` 目录，只保留最近 18 条消息

> **第三步，组装 System Prompt**：动态注入操作系统信息、工作目录、所有可用工具的 JSON Schema、以及行为约束规则。这是 Agent 的『世界观』。

> **第四步，调用 LLM**：通过 `infra/llm.py` 的网关层，根据配置选择模型（默认 deepseek-chat），发送消息并接收响应。

> **第五步，工具调用分支**：这是 ReAct 范式的核心——如果 LLM 返回了 `tool_call`，就执行对应的工具函数，把结果作为新的 ToolMessage 追加回上下文，然后**回到第二步继续循环**；如果没有 tool_call，就输出文本给用户，本轮结束。

> 简单来说：**用户提问 → LLM 思考 → 决定用工具 → 执行 → 结果反馈给 LLM → LLM 继续思考 → 直到直接回答用户**。这就是 ReAct 循环。"

## 第四步：四大子系统逐个拆解

### 子系统1：工具体系 (Tooling System)

**对应文件：** `application/tooling.py`

#### 面试话术

> "工具体系是整个 Agent 的『手』，让 LLM 能真正操作外部世界。我分成了四个工具组：

> **第一组：文件操作工具**——`glob_files`（文件搜索）、`read_file`（读取）、`write_file`（写入）、`edit_file`（精确编辑）、`grep_content`（内容搜索）。这些都通过 `infra/workspace.py` 的安全路径检查，防止越权访问。

> **第二组：Shell 执行工具**——`run_bash` 和 `background_run`。前者是前台命令，有 120 秒超时和危险命令拦截；后者把耗时任务放到后台线程执行，不阻塞主循环。

> **第三组：任务与待办工具**——`task_create`、`task_update`、`task_list`、`task_get`（持久任务板管理）和 `TodoWrite`（轻量待办清单）。这些是 Agent 自我管理、分步执行复杂任务的基础。

> **第四组：协作与通信工具**——`spawn_teammate`（创建持久队友）、`send_message`/`broadcast`（消息通信）、`read_inbox`（读取收件箱）、`shutdown_request`（关闭队友）。这些让 Agent 不再单打独斗，能组建多代理团队。

> **技术亮点**：工具注册采用装饰器模式，把函数的签名自动转换成 LLM 可读的 JSON Schema。这样新增工具只需要写一个普通函数再加个装饰器，不需要改注册代码。同时还有工具结果缓存机制，对读文件等幂等操作做了 LRU 缓存。"

### 子系统2：记忆系统 (Memory System)

**对应文件：** `application/session_memory.py`、`application/compression.py`

#### 面试话术

> "记忆系统解决的是 **『LLM 的有限上下文窗口』** 这个核心矛盾。我设计了三个层次的记忆：

> **第一层：Session Memory（结构化记忆）**——用一个 Markdown 文件 `.memory/session_memory.md` 持久化存储，包含 Session Title、Current State、Task specification、Files and Functions、Workflow、Learnings 六个段落。Agent 在每次主循环开始时，会把这段记忆注入到 System Prompt 里。记忆可以手动更新——比如 project 完成了某个阶段，Agent 会调用工具更新这个文件。

> **第二层：对话压缩 (Compaction)**——针对长对话，我实现了三级压缩策略：
> 1. **微观压缩 (MicroCompact)**：裁剪单条超长工具输出，保留头尾关键信息
> 2. **上下文折叠 (ContextCollapse)**：把早期工具调用改写为摘要，保留最近 8-18 条高保真
> 3. **归档重置 (Transcription)**：超过 80% Token 阈值时，把全部历史归档到 `.transcripts/`，只保留最近的少部分消息，然后继续对话

> **第三层：子代理记忆隔离**——子代理 (`subagent.py`) 有自己的独立消息列表，不会污染主代理的上下文。任务完成后只把摘要返回给主代理。

> **核心思路**：**不是把所有历史都塞给 LLM，而是通过结构化摘要+分层压缩，让 LLM 始终只看到最相关的信息。**"

### 子系统3：多代理协作 (Team Runtime)

**对应文件：** `application/team_runtime.py`、`domain/messaging.py`

#### 面试话术

> "多代理协作是这个项目的亮点之一。我实现了 **『主代理-队友』** 的协作模式：

> **主代理 (Lead Agent)**：负责接收用户指令、规划任务、分配工作。它可以通过 `spawn_teammate` 工具创建持久化的队友代理。

> **队友代理 (Teammate)**：创建后成为一个独立线程运行，有自己的独立循环：读取收件箱 → 处理消息 → 执行任务 → 回复。队友的配置保存在 `.team/config.json` 里，即使进程重启也能恢复。

> **通信机制**：基于文件的收件箱系统。每个队友有自己的收件箱目录 (`.team/inbox/`)，通过 `send_message` 发送消息，`read_inbox` 收取。支持广播 (`broadcast`) 和点对点消息。

> **协作流程**：
> 1. 主代理收到复杂任务 → 创建多个队友（如：一个负责写代码，一个负责测试）
> 2. 主代理通过 send_message 分配子任务
> 3. 队友独立工作，完成后汇报
> 4. 主代理汇总结果，输出给用户

> **亮点设计**：队友的生命周期是持久化的，可以跨越多次用户对话。如果任务耗时长，可以 `shutdown_request` 优雅关闭。"

### 子系统4：领域模型 (Domain Layer)

**对应文件：** `domain/tasks.py`、`domain/todos.py`、`domain/background.py`、`domain/skills.py`

#### 面试话术

> "领域层定义了 Agent 世界里的核心『业务对象』：

> **TaskManager（持久任务板）**：基于文件的任务管理系统，任务持久化在 `.tasks/` 目录里，每个任务一个 JSON 文件。支持依赖关系 (blockedBy/blocks)、状态流转 (pending/in_progress/done)、认领机制 (claim_task)。这是 Agent 做长链路任务规划的基础。

> **TodoManager（轻量待办清单）**：相比 TaskManager 更轻量，内存里的待办列表。Agent 在短链路任务中用 TodoWrite 自我约束，比如『先做 A 再做 B』，确保不会遗漏步骤。

> **BackgroundManager（后台任务管理）**：把耗时的 Shell 命令放到独立线程执行，通过 `background_run` 启动、`check_background` 轮询结果。避免长时间阻塞主循环。

> **SkillLoader（技能加载）**：扫描 `skills/` 目录下的 SKILL.md 文件，把专业知识组织成可加载的知识块。Agent 可以根据任务需要动态加载特定领域的技能。"

## 第五步：关键技术细节（加分项）

> 以下内容是你面试时的加分项——在讲完基础和核心之后，抛出这些技术细节能让面试官眼前一亮。

### 1️⃣ "我为 Shell 执行做了双重安全保护"

> "我们的 Shell 执行层 (`infra/shell.py`) 做了两件事：第一，在 Windows 下统一走 PowerShell 并强制 UTF-8 编码，解决中文乱码的老大难问题；第二，内置危险命令拦截——像 `rm -rf /`、`sudo`、`shutdown` 这些命令在执行前就会被拦截，防止 Agent 失控。"

### 2️⃣ "我实现了精确的文本编辑工具"

> "我们的 `edit_file` 工具不是简单地覆盖文件，而是基于『精确文本替换』——先定位旧文本，再替换为新文本。如果旧文本找不到，工具会给出友好提示。这在 Agent 修改代码时特别有用，比整文件重写更安全可控。"

### 3️⃣ "我用三层压缩策略解决了长上下文问题"

> "LLM 的上下文窗口是有限的。我设计了三级压缩：轻量级裁剪超长工具输出、中等级把早期工具调用改写摘要、重量级归档历史并重置。这三个级别按需逐级触发，而不是一刀切，最大程度保留了最近交互的高保真信息。"

### 4️⃣ "容器化服务管理 (DI 容器)"

> "引入了 `container.py` 作为轻量级的依赖注入容器，统一管理 TaskManager、TodoManager、BackgroundManager、MessageBus、SessionMemoryManager 等服务的生命周期。这让代码更加模块化，测试时也方便 mock 替换。"

### 5️⃣ "多编码兼容的文件读取"

> "在中文 Windows 环境下，文件编码是个大坑。我们的 `read_text_auto` 函数会依次尝试 UTF-8、UTF-8-SIG、GB18030、GBK 四种编码，直到成功。这在处理遗留系统文件时特别有用。"

>

### 🎭 "你在这个项目中遇到了什么技术难点？"

> **（这个回答展示了思考和解决问题的能力）**
>
> "最大的技术难点是 **上下文窗口管理**。Agent 在长时间运行时，消息列表不断膨胀，很快会触及 LLM 的 Token 上限。
>
> 我的解决方案是设计了一套 **三级压缩策略**：
> - 微观层面，对单条超长工具输出做裁剪，保留头尾关键信息
> - 中层，把早期工具调用改写为摘要形式，而非保留完整内容
> - 宏观层面，当上下文超过 80% 阈值时，把历史归档到磁盘，只保留最近的少部分消息
>
> 这个方案的核心权衡是：**在『保留有用信息』和『节省 Token』之间找到平衡点**。太激进的压缩会丢失关键上下文导致 Agent '失忆'，太保守则很快耗尽窗口。我通过可配置的阈值和分级触发机制，让这个平衡可以根据实际场景灵活调整。"

### 🎭 "这个项目和 LangChain 有什么区别？"

> **（展示你的技术判断力）**
>
> "LangChain 是一个通用框架，功能强大但抽象层数多，代码追踪困难。我们的项目追求的是 **『足够结构化，但不过度抽象』**。
>
> 具体来说：我们不引入复杂的 Chain/Graph 抽象，主循环就是一个清晰的 while 循环；工具注册采用简单的装饰器模式，而不是复杂的 Registry 体系；领域模型就是普通的 Python 类加 JSON 持久化，不依赖 ORM 或数据库。
>
> 这带来的好处是：**代码可读性强、调试方便、新增功能成本低**。坏处是：如果需要极复杂的 Agent 工作流编排（比如 DAG 图），可能需要额外开发。所以我们的定位是：**适用于中小规模的 Agent 应用，强调快速迭代和可控性**。"

## 第七步：常见追问与应答策略

### Q1: "你们用的什么大模型？为什么选它？"

> "默认用的是 DeepSeek-chat，通过 .env 文件配置。选择原因有三：一是性价比高，API 价格便宜；二是中文能力优秀，对于我们的项目场景（中文 workspace）很重要；三是支持函数调用 (tool calling) 的标准接口。同时也预留了 Kimi 等其他模型的切换能力。"

### Q2: "Agent 怎么知道自己该调用哪个工具？"

> "这是通过 System Prompt 实现的。我们在每次主循环中，动态把所有工具的 JSON Schema 描述注入到 System Prompt 里。LLM 看到这些描述后，会根据用户意图自主判断该调用哪个工具、传什么参数。返回结果后，我们再根据 LLM 的响应是否有 tool_calls 来决定是继续执行工具还是输出文本。"

### Q3: "任务板和待办清单有什么区别？为什么需要两个？"

> "任务板 (TaskManager) 是持久化的、带依赖关系管理的正式任务系统，适合长链路、多步骤的复杂任务，数据存在磁盘上，重启不丢失。待办清单 (TodoManager) 是轻量的、内存里的自我约束工具，适合短链路、当前轮次内完成的小任务，用完即弃。两者互补——正式任务用任务板跟踪，临时待办用清单提醒。"

### Q4: "多代理协作时怎么保证数据一致性？"

> "我们采取了『最终一致性』的设计思路，而非强一致性。主代理和队友之间通过消息通信，消息是持久化到文件系统的。如果队友崩溃，重启后会从收件箱重新读取未处理的消息。这种设计够用且简单——对于 Agent 场景，不需要像数据库那样的事务保障。"

### Q5: "Session Memory 是怎么工作的？"

> "Session Memory 是一个结构化的 Markdown 文件，包含标题、状态、任务说明、文件函数列表、工作流、经验教训六个段落。每次主循环开始时，这段内容会被注入到 System Prompt 中。Agent 可以通过工具更新这个文件（比如完成某个阶段后更新状态），实现跨轮次的知识持久化。"

### Q6: "这个项目在生产环境部署过吗？遇到过什么问题？"

> "这是实验性项目，但我们已经考虑到了生产化的一些因素：通过 .env 管理配置、支持多模型切换、工具调用有超时和错误处理、Shell 执行有安全拦截。如果要上生产，主要考虑加强的是：更完善的日志和监控、LLM 调用 retry 机制、以及更细粒度的权限控制。"
